# 权限模型 V1

## 一、设计目标

第一阶段采用 **基于角色的访问控制（RBAC）**，不引入多租户与细粒度资源 ACL。所有 API 在 `apps/api` 中间件层统一校验。

## 二、角色定义

| 角色 | 代码 | 说明 |
| ---- | ---- | ---- |
| 管理员 | ADMIN | 系统配置、用户管理、全部业务 |
| 运营 | OPERATOR | 选题、内容、生成、发布（不含终审配置） |
| 审核 | REVIEWER | 审核中心、只读内容详情 |
| 分析 | ANALYST | 数据复盘、看板只读 |

对应数据库枚举：`UserRole`（见《数据库设计文档 V1》）。

## 三、权限矩阵

图例：✓ 允许 · R 只读 · — 禁止

### 3.1 模块权限

| 模块 / 操作 | ADMIN | OPERATOR | REVIEWER | ANALYST |
| ----------- | ----- | -------- | -------- | ------- |
| 用户管理 | ✓ | — | — | — |
| 系统设置 | ✓ | — | — | — |
| 平台账号（增删改） | ✓ | ✓ | R | R |
| 账号画像（增删改） | ✓ | ✓ | R | R |
| 选题（增删改） | ✓ | ✓ | R | R |
| 内容资产（增删改） | ✓ | ✓ | R | R |
| AI 生成 / Agent 执行 | ✓ | ✓ | — | — |
| Prompt / Agent 配置 | ✓ | R | — | — |
| IMA 搜索 | ✓ | ✓ | — | — |
| 提交审核 | ✓ | ✓ | — | — |
| 审核通过 / 驳回 | ✓ | R | ✓ | — |
| 创建发布任务 | ✓ | ✓ | — | — |
| 执行发布 | ✓ | ✓ | — | — |
| 数据同步 / 复盘报告生成 | ✓ | R | — | ✓ |
| Dashboard | ✓ | ✓ | R | ✓ |

说明：

- OPERATOR 可「提交审核」，不可「终审」自己提交的内容（见 3.2）。
- REVIEWER 对 Prompt/Agent 仅查看，防止改动生成策略。

### 3.2 业务规则

1. **审核隔离**：`REVIEWER` 不能审核 `reviewerId === 当前用户` 且 `createdBy === 当前用户` 的同一条内容（自己创建的自己不能审）。
2. **发布前置**：仅 `ContentVersion.status === APPROVED` 可创建 `PublishingTask`。
3. **账号归属**：`OPERATOR` 默认只能操作 `platform_accounts.ownerId === 当前用户` 的账号；`ADMIN` 可操作全部。
4. **配置敏感项**：模型 API Key、TurboPush 凭证仅存 `system_configs`，仅 `ADMIN` 可读写。

## 四、API 鉴权约定

| 项 | 约定 |
| --- | --- |
| 认证 | JWT Bearer，`Authorization: Bearer <token>` |
| 登录 | `POST /api/auth/login` |
| 当前用户 | `GET /api/auth/me` |
| 权限失败 | HTTP 403，`{ "code": 403, "message": "forbidden", "data": null }` |
| 未登录 | HTTP 401 |

中间件伪代码：

```ts
function requireRoles(...roles: UserRole[]) {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) return res.status(403).json(forbidden())
    next()
  }
}
```

## 五、与 TurboPush 权限的关系

| 能力 | 策略 |
| ---- | ---- |
| TurboPush 原用户/团队 | 直接复用其登录与会话；中台 API 映射为本地 `users` |
| TurboPush 账号授权 | 通过 Adapter 同步；中台 `OPERATOR` 权限 ≤ TurboPush 账号可见范围 |
| 发布操作 | 中台校验 RBAC 后，Adapter 使用 TurboPush 侧已授权账号执行 |
